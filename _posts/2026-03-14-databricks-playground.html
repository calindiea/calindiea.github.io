Delta Lake Internals

Transaction log, file layout, and how snapshots reconstruct table state.

📒 Transaction Log (_delta_log)

Every mutation (INSERT, UPDATE, DELETE, MERGE) creates a new JSON commit file in _delta_log/. Each commit is an atomic unit containing actions:

  • add — new Parquet file written
  • remove — file logically deleted
  • txn — idempotent streaming ID
  • metaData — schema/partition changes
  • protocol — reader/writer version

📸 Snapshots & Checkpoints

To read a table, Delta replays log entries to build a snapshot — the set of "active" files at a version.

Every 10 commits, a Parquet checkpoint is written so readers don't replay from version 0. The _last_checkpoint file points to the latest checkpoint.

Partitioning

How Hive-style partitioning lays out files on disk and enables partition pruning.

📁 Partition Layout

When you PARTITIONED BY (col), each distinct value of col gets its own directory. Files within a partition only contain rows matching that value.

Partition columns are not stored inside Parquet — they're inferred from the directory path. This saves storage but means partition columns have no min/max stats for data skipping.

🔍 Partition Pruning

When a query has WHERE partition_col = 'X', the engine reads only the directories matching X, skipping all others entirely — zero I/O on pruned partitions.

High-cardinality columns (e.g., user_id) create millions of tiny files → "small file problem." Choose columns with low-to-medium cardinality (date, region, status).

Z-Order (Multi-Dimensional Clustering)

Co-locate related data within Parquet files so data skipping eliminates more files per query.

🔀 Z-Order Curve

Z-Ordering interleaves the bits of multiple column values to produce a single sort key. Rows with similar values across all Z-Order columns land in the same file.

This creates tight min/max ranges in file-level statistics, enabling Delta's data skipping to prune files even when filtering on non-partition columns.

📊 Data Skipping

Each Parquet file footer stores min/max stats per column. When Z-Ordered, the min/max ranges per file are narrow → more files can be skipped.

OPTIMIZE table ZORDER BY (col1, col2) — rewrites data files. This is an expensive operation (full rewrite) but pays off with faster reads.

Liquid Clustering

Databricks' next-gen clustering — automatic, incremental, and column-changeable at any time.

💧 How It Works

Liquid Clustering uses Hilbert space-filling curves (better locality than Z-curves) and applies clustering incrementally — only newly written or unclustered files are reorganized.

  • Defined at table creation: CLUSTER BY (col1, col2)
  • Columns can be changed later with ALTER TABLE ... CLUSTER BY
  • Triggered automatically by OPTIMIZE (no ZORDER BY clause needed)
  • Uses eager clustering in some write paths

OPTIMIZE / VACUUM / ANALYZE

Table maintenance commands — compaction, cleanup, and statistics.

🔧 OPTIMIZE (Compaction)

Combines small files into larger ones (target ~1 GB). Can also apply Z-Ordering or Liquid Clustering.

OPTIMIZE table WHERE date = '2024-01-01' — you can scope OPTIMIZE to specific partitions to reduce cost.

🧹 VACUUM

Physically deletes files that are no longer referenced by any active version and are older than the retention period (default: 7 days).

Warning: After VACUUM, you cannot time travel to versions that relied on the deleted files. Set retention carefully.

📈 ANALYZE TABLE

Computes column-level statistics (distinct count, null count, min, max, avg length) that the optimizer uses for cost-based optimization (CBO) — better join strategies, better broadcast decisions.

ANALYZE TABLE t COMPUTE STATISTICS FOR ALL COLUMNS

✅ When to Use

  • OPTIMIZE: After bulk loads, streaming micro-batches, or many small appends. Schedule daily/hourly.
  • VACUUM: Run periodically (daily/weekly) to reclaim storage. After confirming no active readers need old versions.
  • ANALYZE: After major data changes, before complex analytical queries with expensive joins.
